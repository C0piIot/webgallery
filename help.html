What this is

webgallery is a Progressive Web App that doubles as a personal photo and video gallery and a backup tool. It runs entirely in your browser with no server of its own — it talks directly to an S3-compatible bucket you provide. Targeted at Chrome on Android and desktop.

How storage works

An S3 bucket is a remote folder served by a provider over an HTTPS API. You authenticate with an access-key + secret pair. The provider bills you for bytes stored and bytes transferred.

S3-compatible means a provider implements the same API; any of them works with this app. Pick one that fits your budget and trust requirements.

Security model

This app is fully client-side. The HTML and JS come from a public GitHub Pages site under the maintainer's account. If the maintainer pushes malicious JS — or if their account is hacked — the new code runs in your browser on the next load and can read your bucket credentials out of IndexedDB.

Content Security Policy partially mitigates this (the app loads no third-party origins, all dependencies are vendored) but does not prevent first-party JS from misbehaving. You're trusting whoever controls the Pages source.

Mitigations, strongest first

  1. Fork and self-host. Fork the repo, enable Pages on the fork, point your devices at your fork's URL. Review upstream commits before merging. You're now trusting yourself plus GitHub's infrastructure.
  2. Least-privilege credentials. Create a key scoped to one bucket and one prefix, granting only the S3 actions this app actually uses (see the example policy below). Rotate the key if anything ever feels wrong.
  3. Bucket versioning + tight CORS. The app hard-deletes — versioning is your only undo. Restrict CORS to the origin you serve the app from.

Example IAM policy (AWS-flavored)

Replace your-bucket and your-prefix. Other providers use similar JSON shapes. 

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject",
        "s3:AbortMultipartUpload"
      ],
      "Resource": "arn:aws:s3:::your-bucket/your-prefix/*"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::your-bucket",
      "Condition": {
        "StringLike": { "s3:prefix": ["your-prefix/*"] }
      }
    }
  ]
}

Example CORS configuration

Replace the origin with where you serve the app from (your GitHub Pages URL or your fork's URL). 

[
  {
    "AllowedOrigins": ["https://your-username.github.io"],
    "AllowedMethods": ["GET", "PUT", "HEAD", "DELETE", "POST"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]

Standing recommendations

  • Use HTTPS-only endpoints in production.
  • Don't reuse the bucket for other apps — a leaked key escalates sideways.
  • Watch your provider's billing dashboard for traffic anomalies.
